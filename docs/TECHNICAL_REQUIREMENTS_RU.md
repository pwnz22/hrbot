# Документ технических требований
## HRBot - Система автоматизации HR процессов в Telegram

**Версия:** 1.0
**Последнее обновление:** 2026-03-06
**Статус:** Активная разработка

---

## 1. Обзор системы

### 1.1 Назначение
HRBot — это система автоматизации HR процессов на базе Telegram, которая оптимизирует рекрутинговые процессы путем автоматического парсинга откликов на вакансии из Gmail, анализа резюме с использованием ИИ и предоставления рекрутерам структурированной информации о кандидатах через интерактивный интерфейс бота.

### 1.2 Область применения
- Автоматический парсинг электронной почты из нескольких аккаунтов Gmail
- Анализ резюме и оценка кандидатов с использованием ИИ
- Ролевое управление доступом для членов команды рекрутинга
- Управление жизненным циклом заявок
- Возможности экспорта данных для отчетности

### 1.3 Целевые пользователи
- **Администраторы**: Системные администраторы с полным доступом
- **Модераторы**: HR-специалисты, управляющие заявками
- **Пользователи**: Ограниченный доступ, регистрация ожидает одобрения

## 2. Функциональные требования

### 2.1 Аутентификация и авторизация

#### ФТ-2.1.1 Регистрация пользователей
- Система ДОЛЖНА автоматически создавать записи пользователей при первом взаимодействии
- Система ДОЛЖНА назначать роль USER по умолчанию новым регистрациям
- Система ДОЛЖНА сохранять Telegram user ID, имя пользователя, имя и фамилию

#### ФТ-2.1.2 Ролевое управление доступом
- Система ДОЛЖНА поддерживать три уровня ролей: USER, MODERATOR, ADMIN
- Система ДОЛЖНА обеспечивать проверку прав доступа на уровне обработчиков
- Система ДОЛЖНА предотвращать несанкционированный доступ к защищенным командам

#### ФТ-2.1.3 Управление ролями
- ADMIN пользователи ДОЛЖНЫ иметь возможность повышать/понижать права пользователей
- Система ДОЛЖНА вести журнал аудита изменений ролей
- Система ДОЛЖНА поддерживать инициализацию ролей на уровне базы данных

### 2.2 Интеграция с Gmail

#### ФТ-2.2.1 Поддержка множественных аккаунтов
- Система ДОЛЖНА поддерживать несколько аккаунтов Gmail одновременно
- Система ДОЛЖНА поддерживать отдельные OAuth токены для каждого аккаунта
- Система ДОЛЖНА разрешать фильтрацию по email отправителя для каждого аккаунта

#### ФТ-2.2.2 OAuth аутентификация
- Система ДОЛЖНА реализовать поток авторизации OAuth 2.0
- Система ДОЛЖНА безопасно хранить refresh-токены
- Система ДОЛЖНА автоматически обновлять истекшие access-токены
- Система ДОЛЖНА корректно обрабатывать истечение срока действия токенов

#### ФТ-2.2.3 Парсинг электронной почты
- Система ДОЛЖНА опрашивать аккаунты Gmail через настраиваемые интервалы
- Система ДОЛЖНА извлекать имя кандидата, email, телефон из содержимого письма
- Система ДОЛЖНА обнаруживать и загружать вложения резюме (.pdf, .docx)
- Система ДОЛЖНА разбирать сопроводительные письма и сообщения кандидатов
- Система ДОЛЖНА связывать заявки с вакансиями на основе темы письма
- Система ДОЛЖНА создавать новые записи вакансий для нераспознанных позиций

#### ФТ-2.2.4 Управление аккаунтами
- ADMIN ДОЛЖЕН иметь возможность добавлять/удалять аккаунты Gmail
- ADMIN ДОЛЖЕН настраивать фильтры email отправителей для каждого аккаунта
- Система ДОЛЖНА проверять OAuth учетные данные перед активацией
- Система ДОЛЖНА отображать статус и статистику аккаунта

### 2.3 Обработка резюме

#### ФТ-2.3.1 Извлечение содержимого документов
- Система ДОЛЖНА извлекать текст из PDF файлов используя PyPDF2
- Система ДОЛЖНА извлекать текст из DOCX файлов используя python-docx
- Система ДОЛЖНА корректно обрабатывать поврежденные или некорректные файлы
- Система ДОЛЖНА логировать ошибки извлечения с деталями

#### ФТ-2.3.2 Анализ с использованием ИИ
- Система ДОЛЖНА интегрироваться с Google Gemini API
- Система ДОЛЖНА генерировать структурированные резюме кандидатов, включая:
  - Ключевые сильные стороны и квалификации
  - Список технических и гибких навыков
  - Стаж работы в годах
  - Образование
  - Флаги рисков и проблемные моменты
  - Предлагаемые вопросы для собеседования с ответами

#### ФТ-2.3.3 Хранение резюме
- Система ДОЛЖНА сохранять сгенерированные ИИ резюме в базе данных
- Система ДОЛЖНА связывать резюме с записями заявок
- Система ДОЛЖНА поддерживать регенерацию резюме

### 2.4 Управление заявками

#### ФТ-2.4.1 Список заявок
- Система ДОЛЖНА отображать заявки, сгруппированные по вакансиям
- Система ДОЛЖНА показывать количество необработанных заявок
- Система ДОЛЖНА поддерживать фильтрацию по статусу обработки
- Система ДОЛЖНА использовать пагинацию для больших наборов данных

#### ФТ-2.4.2 Детали заявки
- Система ДОЛЖНА отображать полную информацию о кандидате
- Система ДОЛЖНА показывать сгенерированные ИИ резюме
- Система ДОЛЖНА предоставлять ссылки для скачивания резюме
- Система ДОЛЖНА отображать источник заявки (email/Telegram)

#### ФТ-2.4.3 Управление статусами
- MODERATOR ДОЛЖЕН иметь возможность отмечать заявки как обработанные/необработанные
- Система ДОЛЖНА отслеживать временные метки обработки
- Система ДОЛЖНА поддерживать заметки и описания обработки

#### ФТ-2.4.4 Удаление заявок
- MODERATOR ДОЛЖЕН иметь возможность удалять заявки
- Система ДОЛЖНА реализовать мягкое удаление с временной меткой deleted_at
- Система ДОЛЖНА исключать удаленные записи из стандартных запросов

### 2.5 Управление вакансиями

#### ФТ-2.5.1 CRUD операции с вакансиями
- ADMIN ДОЛЖЕН иметь возможность создавать вакансии вручную
- Система ДОЛЖНА автоматически создавать вакансии из тем писем
- ADMIN ДОЛЖЕН иметь возможность редактировать детали вакансии
- ADMIN ДОЛЖЕН иметь возможность удалять вакансии (каскадно с заявками)

#### ФТ-2.5.2 Отображение вакансий
- Система ДОЛЖНА отображать список всех активных вакансий
- Система ДОЛЖНА показывать количество заявок на каждую вакансию
- Система ДОЛЖНА отображать источник вакансии (email/ручное создание)

### 2.6 Экспорт данных

#### ФТ-2.6.1 Экспорт в Excel
- MODERATOR ДОЛЖЕН иметь возможность экспортировать заявки в Excel
- Система ДОЛЖНА включать все поля кандидата
- Система ДОЛЖНА включать сгенерированные ИИ резюме
- Система ДОЛЖНА поддерживать фильтрацию экспорта (все/необработанные)
- Система ДОЛЖНА генерировать имена файлов с временными метками

#### ФТ-2.6.2 Формат экспорта
- Файлы ДОЛЖНЫ быть в формате .xlsx
- Столбцы ДОЛЖНЫ включать: Имя, Email, Телефон, Вакансия, Статус, Навыки, Опыт, Резюме, Вопросы для собеседования
- Система ДОЛЖНА корректно обрабатывать Unicode символы

### 2.7 Пользовательский интерфейс

#### ФТ-2.7.1 Интерфейс команд
- Система ДОЛЖНА поддерживать следующие команды:
  - `/start` - Инициализация взаимодействия с ботом
  - `/parse_now` - Запуск ручного парсинга электронной почты
  - `/manage_accounts` - Управление аккаунтами Gmail
  - `/list_users` - Отображение списка пользователей
  - `/export_all` - Экспорт всех заявок
  - `/export_unprocessed` - Экспорт только необработанных

#### ФТ-2.7.2 Встроенные клавиатуры
- Система ДОЛЖНА использовать встроенные клавиатуры для навигации
- Система ДОЛЖНА поддерживать обработку callback-запросов
- Система ДОЛЖНА реализовать иерархические структуры меню
- Система ДОЛЖНА предоставлять кнопки отмены/назад

#### ФТ-2.7.3 Форматирование сообщений
- Система ДОЛЖНА форматировать сообщения используя Markdown
- Система ДОЛЖНА отображать данные в структурированном формате
- Система ДОЛЖНА сокращать длинный текст с кнопками "Показать больше"
- Система ДОЛЖНА использовать эмодзи для визуальной ясности

## 4. Техническая архитектура

### 4.1 Компоненты системы

#### 4.1.1 Слой бота
- **Фреймворк**: Aiogram 3.10
- **Обязанности**:
  - Обработка команд
  - Обработка callback-запросов
  - Управление взаимодействием с пользователем
  - Планирование фоновых задач

#### 4.1.2 Слой сервисов
- **Компоненты**:
  - `GeminiService`: Анализ резюме с помощью ИИ
  - `ResumeSummaryService`: Оркестрация генерации резюме
  - `DocumentTextExtractor`: Обработка файлов
- **Обязанности**:
  - Инкапсуляция бизнес-логики
  - Коммуникация с внешними API
  - Трансформация данных

#### 4.1.3 Слой данных
- **ORM**: SQLAlchemy 2.0
- **Драйвер**: aiosqlite (async)
- **Миграции**: Alembic
- **Обязанности**:
  - Сохранение данных
  - Оптимизация запросов
  - Версионирование схемы

#### 4.1.4 Слой интеграции
- **Gmail API**: Получение и парсинг электронной почты
- **Gemini API**: Анализ с использованием ИИ
- **Telegram Bot API**: Взаимодействие с пользователем

### 4.2 Архитектурные паттерны

#### 4.2.1 Архитектура слоя сервисов
- Бизнес-логика ДОЛЖНА быть изолирована в классах сервисов
- Обработчики ДОЛЖНЫ делегировать задачи сервисам
- Сервисы НЕ ДОЛЖНЫ зависеть от фреймворка бота

#### 4.2.2 Внедрение зависимостей
- Сервисы ДОЛЖНЫ получать зависимости через конструктор
- Сессии базы данных ДОЛЖНЫ инжектиться в сервисы
- Конфигурация ДОЛЖНА инжектиться через окружение

#### 4.2.3 Паттерн Middleware
- Сквозные задачи ДОЛЖНЫ использовать middleware
- Проверка ролей ДОЛЖНА быть реализована как middleware
- Обработка ошибок ДОЛЖНА использовать middleware

#### 4.2.4 Async-first дизайн
- Все I/O операции ДОЛЖНЫ использовать async/await
- Операции с базой данных ДОЛЖНЫ быть асинхронными
- API вызовы ДОЛЖНЫ быть асинхронными
- Файловые операции ДОЛЖНЫ использовать aiofiles где применимо

### 4.3 Технологический стек

#### 4.3.1 Среда выполнения

- asyncio event loop

#### 4.3.2 Фреймворки и библиотеки
- `aiogram 3.10`: Фреймворк Telegram бота

#### 4.3.3 Внешние сервисы

#### 4.3.4 Инструменты разработки
- Docker & Docker Compose

## 5. Модели данных и схема

### 5.1 Основные сущности

#### 5.1.1 TelegramUser
```
- id: Integer (PK, автоинкремент)
- telegram_id: BigInteger (уникальный, индексируется)
- username: String (nullable)
- first_name: String (nullable)
- last_name: String (nullable)
- role: Enum(USER, MODERATOR, ADMIN)
- created_at: DateTime
- updated_at: DateTime
```

**Связи:**
- 1:N с GmailAccount (владелец)

#### 5.1.2 GmailAccount
```
- id: Integer (PK, автоинкремент)
- email: String (уникальный)
- credentials_path: String
- token_path: String
- added_by_user_id: Integer (FK к TelegramUser)
- is_active: Boolean
- sender_emails: Text (JSON массив)
- created_at: DateTime
- updated_at: DateTime
```

**Связи:**
- N:1 с TelegramUser (добавлен пользователем)
- 1:N с Vacancy

#### 5.1.3 Vacancy
```
- id: Integer (PK, автоинкремент)
- title: String (индексируется)
- description: Text (nullable)
- source: Enum(email, telegram)
- gmail_account_id: Integer (FK к GmailAccount, nullable)
- created_at: DateTime
- updated_at: DateTime
```

**Связи:**
- N:1 с GmailAccount
- 1:N с Application

#### 5.1.4 Application
```
- id: Integer (PK, автоинкремент)
- vacancy_id: Integer (FK к Vacancy)
- name: String
- email: String (индексируется)
- phone: String (nullable)
- resume_path: String (nullable)
- cover_letter: Text (nullable)
- source: Enum(email, telegram)
- message_id: Integer (nullable)
- summary: Text (nullable)
- skills: Text (JSON массив, nullable)
- experience_years: String (nullable)
- education: Text (nullable)
- red_flags: Text (nullable)
- interview_questions: Text (nullable)
- is_processed: Boolean (по умолчанию=False)
- processing_description: Text (nullable)
- processed_at: DateTime (nullable)
- deleted_at: DateTime (nullable)
- created_at: DateTime
- updated_at: DateTime
```

**Связи:**
- N:1 с Vacancy

### 5.2 Требования к базе данных

#### 5.2.1 Стратегия индексирования
- Индекс на `telegram_users.telegram_id` для O(1) поиска пользователя

## 6. Интеграции с внешними API

### 6.1 Telegram Bot API

#### 6.1.1 Аутентификация
- Токен бота ДОЛЖЕН храниться в переменной окружения BOT_TOKEN
- Токен ДОЛЖЕН использовать HTTPS webhook или long-polling

#### 6.1.2 Использование API
- sendMessage для текстовых ответов
- sendDocument для скачивания файлов
- editMessageText для динамических обновлений
- answerCallbackQuery для взаимодействия с встроенными клавиатурами

#### 6.1.3 Лимиты скорости
- Максимум 30 сообщений в секунду на чат
- Максимум 20 сообщений в минуту в одну группу
- Реализовать экспоненциальную задержку при ошибках лимита скорости

### 6.2 Google Gmail API

#### 6.2.1 Аутентификация
- OAuth 2.0 с офлайн доступом
- Области доступа: `gmail.readonly`
- Файл учетных данных: `credentials.json` (игнорируется git)
- Хранилище токенов: `gmail_tokens/{email}.json`

#### 6.2.2 Использование API
- `users.messages.list`: Получение ID сообщений
- `users.messages.get`: Получение деталей сообщения
- `users.messages.attachments.get`: Скачивание вложений
- Фильтры запросов: `from:{sender_email} is:unread`

#### 6.2.3 Обработка ошибок
- Обработка 401 Unauthorized (обновление токена)
- Обработка 403 Forbidden (недостаточно прав)
- Обработка 429 Rate Limit (экспоненциальная задержка)
- Обработка 500 Server Error (логика повторных попыток)

#### 6.2.4 Лимиты скорости
- 250 единиц квоты в секунду на пользователя
- 1 миллиард единиц квоты в день
- Получение сообщения: 5 единиц квоты на запрос

### 6.3 Google Gemini API

#### 6.3.1 Аутентификация
- API ключ хранится в переменной окружения GEMINI_API_KEY
- Использование Python SDK `google.generativeai`

#### 6.3.2 Конфигурация модели
- Модель: `gemini-1.5-flash` или `gemini-1.5-pro`
- Temperature: 0.7
- Max tokens: 2048
- Настройки безопасности: BLOCK_MEDIUM_AND_ABOVE

#### 6.3.3 Разработка промптов
- Структурированные промпты для консистентного вывода
- Формат ответа JSON с определенной схемой
- Включение текста резюме и контекста позиции
- Запрос конкретных полей: summary, skills, experience, red_flags, questions

#### 6.3.4 Обработка ошибок
- Обработка 400 Bad Request (некорректный промпт)
- Обработка 429 Rate Limit (экспоненциальная задержка)
- Обработка 500 Server Error (повтор с резервной моделью)
- Таймаут: 30 секунд на запрос

#### 6.3.5 Управление затратами
- Кэширование текста резюме для избежания дублирования анализа
- Использование flash модели для стандартных резюме
- Использование pro модели только для сложных резюме
- Мониторинг ежемесячного использования токенов

## 15. Глоссарий

**Заявка (Application)**: Заявка кандидата на работу, включая резюме и контактную информацию

**Вакансия (Vacancy)**: Открытая позиция или вакансия для найма

**Обработка (Processing)**: Пометка заявки как просмотренной командой HR

**Мягкое удаление (Soft Delete)**: Пометка записей как удаленных без удаления из базы данных

**OAuth токен**: Временные учетные данные для доступа к Gmail API

**Callback Query**: Взаимодействие с кнопкой встроенной клавиатуры Telegram

**Слой сервисов (Service Layer)**: Компоненты бизнес-логики, изолированные от слоя представления
